## 1. Introduction
In complex GPU clusters, the bridge between physical silicon and Kubernetes workloads is often a "black box." When an engineer identifies a performance bottleneck or a hardware error on a specific MIG instance (e.g., `GI 4`, `CI 0`), they need to know exactly which Pod is responsible.

This module demonstrates how **DRA ResourceSlices** and **CDI (Container Device Interface)** provide full traceability from the physical hardware IDs to the container level.

---

## 2. Core Concepts

### 2.1 The Traceability Loop
1.  **Device Discovery**: DRA publishes `ResourceSlices` with physical profile names (e.g., 3g.20gb).
2.  **Infrastructure Bindings**: `ResourceClaims` record the `ReservedFor` metadata of the Pod.
3.  **CDI Mapping**: The CDI specification translates K8s device requests into specific Linux device paths (e.g., `/dev/nvidia-caps/nvidia-cap144`).

### 2.2 Observability Advantage
Unlike generic device plugins, DRA stores the **Physical Location** of the slice in the API, allowing external observability tools to map metrics (Power, Temperature, SM Util) directly to the tenant without guessing.
