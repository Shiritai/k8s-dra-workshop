# Module 11.1: MIG MPS SM Limits

**Status**: ✅ **Verified** (K8s v1.34.0, NVIDIA A100-PCIE-40GB x2, DRA driver v25.8.1)

## 1. Introduction

This module tests whether the DRA scheduler and MPS runtime enforce SM (Streaming Multiprocessor) thread percentage limits on MIG devices. We compare two approaches:

- **Server-side**: `defaultActiveThreadPercentage` in the ResourceClaim's `MigDeviceConfig` (daemon-level ceiling)
- **Client-side**: `CUDA_MPS_ACTIVE_THREAD_PERCENTAGE` environment variable (per-pod limit)

### Cluster Configuration

All tests run on a cluster with **3 MIG devices** across 2 physical A100-PCIE-40GB GPUs:

| Device Name | Parent GPU | MIG Profile | SMs | Memory |
|---|---|---|---|---|
| `gpu-0-mig-9-4-4` | GPU-0 | 3g.20gb | 42 | ~20096 MiB |
| `gpu-0-mig-5-0-4` | GPU-0 | 4g.20gb | 56 | ~20096 MiB |
| `gpu-1-mig-0-0-8` | GPU-1 | 7g.40gb | 98 | ~40320 MiB |

All devices are published via a single ResourceSlice with `deviceClassName: mig.nvidia.com`.

## 2. Test 11-1a: Server-Side SM Limit (Daemon-Level)

| Property | Value |
|---|---|
| File | `manifests/module11/11-1a-sm-server-side.yaml` |
| CUDA source | `manifests/module11/cuda/fma_bench.cu` |
| Setup | 1 shared ResourceClaim (`m11-1a-claim`) with `defaultActiveThreadPercentage: 50`, 3 Pods sharing the same MIG device |
| Target | MIG device via `mig.nvidia.com` |
| Server-side SM ceiling | 50% (daemon-level, shared by all clients) |

### Design

A single ResourceClaim (`m11-1a-claim`) with `defaultActiveThreadPercentage: 50`. Three pods share the same MIG device (3g.20gb, 42 SMs) via a single MPS daemon. This design verifies that **server-side `defaultActiveThreadPercentage` is a per-daemon global ceiling, not a per-client limit**. All pods should see the same 50% SM ceiling and produce roughly equal GFLOPS throughput.

### Expected Behavior

- All 3 pods share the same ResourceClaim and are allocated to the same MIG device.
- The MPS daemon applies 50% thread percentage as a global ceiling (50% × 42 SMs ≈ 20 SMs).
- **All 3 pods should see the same SM ceiling (~20 SMs) and produce roughly equal GFLOPS throughput.**

### Actual Results

- All 3 pods were successfully scheduled and entered Running state.
- The single claim `m11-1a-claim` was allocated to `gpu-0-mig-9-4-4` (3g.20gb, 42 SMs).

**Compute Benchmark Results** (FMA stress test: N=16M, INNER_ITERS=64, 200 launches):

| Pod | Claim | MIG Device | Server-Side Ceiling | SMs | Effective SMs | Throughput (GFLOPS) |
|---|---|---|---|---|---|---|
| Pod 1 | `m11-1a-claim` | `gpu-0-mig-9-4-4` (3g.20gb) | 50% | 42 | 20 | 2289 |
| Pod 2 | `m11-1a-claim` | `gpu-0-mig-9-4-4` (3g.20gb) | 50% | 42 | 20 | 2299 |
| Pod 3 | `m11-1a-claim` | `gpu-0-mig-9-4-4` (3g.20gb) | 50% | 42 | 20 | 2476 |

All pods see the same effective SM count (20). The GFLOPS variance is within ~8%, which is normal run-to-run variation.

### Conclusions

1. **`defaultActiveThreadPercentage` is effective**: 50% × 42 SMs = ~20 SMs. All 3 pods report `multiProcessorCount` = 20, and throughput is consistent. This confirms the server-side SM ceiling works.
2. **Server-side config is per-daemon (per-claim), not per-client (per-pod).** All pods sharing the same claim see the exact same SM ceiling. It is not possible to assign different SM percentages to different pods via server-side config.
3. **For per-pod SM differentiation, use the client-side `CUDA_MPS_ACTIVE_THREAD_PERCENTAGE` env var** (see 11-1b).

## 3. Test 11-1b: Client-Side SM Limit (CUDA_MPS_ACTIVE_THREAD_PERCENTAGE)

| Property | Value |
|---|---|
| File | `manifests/module11/11-1b-sm-client-side.yaml` |
| CUDA source | `manifests/module11/cuda/fma_bench.cu` (shared with 11-1a) |
| Setup | 1 ResourceClaim (MPS, no `defaultActiveThreadPercentage` — daemon ceiling = 100%), 3 Pods with client-side env var |
| Target | MIG device via `mig.nvidia.com` |
| Pod thread percentages | Pod 1: 10%, Pod 2: 30%, Pod 3: 50% |

### Design

This test uses the **client-side** env var `CUDA_MPS_ACTIVE_THREAD_PERCENTAGE` to enforce per-pod SM limits. Three pods share one MIG device via a single ResourceClaim. The claim has **no** `defaultActiveThreadPercentage` (daemon ceiling defaults to 100%), so all SM control is purely client-side. Contrast with 11-1a: 11-1a uses server-side `defaultActiveThreadPercentage: 50` as a global ceiling (all pods see the same ceiling), while this test uses client-side env vars to assign different percentages (10/30/50%) to each pod, verifying per-pod differentiation.

### Actual Results

FMA stress test on `gpu-0-mig-9-4-4` (3g.20gb, 42 SMs):

| Pod | Thread % | SMs | Throughput (GFLOPS) |
|---|---|---|---|
| Pod 1 | 10% | 4 | 507.03 |
| Pod 2 | 30% | 12 | 1437.87 |
| Pod 3 | 50% | 20 | 2331.31 |

### Throughput Ratio Analysis

| Comparison | Thread % ratio | SM ratio | GFLOPS ratio | Scaling efficiency |
|---|---|---|---|---|
| Pod 2 / Pod 1 | 3.0× (30/10) | 3.0× (12/4) | **2.84×** (1437.87/507.03) | ~95% of SM ratio |
| Pod 3 / Pod 1 | 5.0× (50/10) | 5.0× (20/4) | **4.60×** (2331.31/507.03) | ~92% of SM ratio |
| Pod 3 / Pod 2 | 1.67× (50/30) | 1.67× (20/12) | **1.62×** (2331.31/1437.87) | ~97% of SM ratio |

Throughput scales near-linearly with SM count. The slight sub-linear scaling (~92–97% efficiency) is expected due to memory bandwidth becoming the bottleneck as SM count increases — the FMA kernel is compute-heavy but still incurs memory traffic for the initial load and final store.

### Conclusions

1. **`CUDA_MPS_ACTIVE_THREAD_PERCENTAGE` works on MIG**: The client-side env var successfully enforces per-client SM limits on MIG devices, providing true per-pod resource isolation. Each pod's `cudaGetDeviceProperties().multiProcessorCount` reflects the limited SM count (4, 12, 20), not the full MIG slice's 42 SMs.
2. **Throughput is near-linearly proportional to SM count**: Pod 3 (50%, 20 SMs) achieves 4.60× the throughput of Pod 1 (10%, 4 SMs), closely matching the 5× SM ratio. This confirms the SM limit directly controls compute capacity with high efficiency.
3. **Client-side env var enables per-pod differentiation**: Server-side `defaultActiveThreadPercentage` (11-1a) can only set a global ceiling (all pods share the same ceiling), while the client-side env var can assign different SM percentages to each pod on the same shared device. This is the recommended approach for per-pod compute isolation on MIG.

## 5. Summary

| Approach | Granularity | Enforced on MIG? | Characteristics | Use Case |
|---|---|---|---|---|
| Server-side `defaultActiveThreadPercentage` (11-1a) | Per-daemon (per-claim) | **Yes** — effective but uniform (global cap) | All pods under the same claim see the same ceiling | Global SM ceiling control (all clients share one ceiling) |
| Client-side `CUDA_MPS_ACTIVE_THREAD_PERCENTAGE` (11-1b) | Per-client (per-pod) | **Yes** — effective and differentiated (per-pod) | Each pod can be assigned a different percentage | Production per-pod SM differentiation |

**Key insight**: Server-side `defaultActiveThreadPercentage` is effective, but it is a per-daemon global ceiling — all pods under the same claim see the exact same SM ceiling (11-1a: all 3 pods get 20 SMs / ~2300 GFLOPS). Client-side `CUDA_MPS_ACTIVE_THREAD_PERCENTAGE` enables true per-pod differentiation (11-1b: 10%=4 SMs/505 GFLOPS, 30%=12 SMs/1434 GFLOPS, 50%=20 SMs/2334 GFLOPS). If you need to assign different SM percentages to different pods, client-side is the only viable approach.
