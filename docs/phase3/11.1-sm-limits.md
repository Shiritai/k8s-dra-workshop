# Module 11.1: MIG MPS SM Limits

**Status**: ✅ **Verified** (K8s v1.34.0, NVIDIA A100-PCIE-40GB x2, DRA driver v25.8.1)

## 1. Introduction

This module tests whether the DRA scheduler and MPS runtime enforce SM (Streaming Multiprocessor) thread percentage limits on MIG devices. We compare two approaches:

- **Server-side**: `defaultActiveThreadPercentage` in the ResourceClaim's `MigDeviceConfig` (daemon-level ceiling)
- **Client-side**: `CUDA_MPS_ACTIVE_THREAD_PERCENTAGE` environment variable (per-pod limit)

### Cluster Configuration

All tests run on a cluster with **3 MIG devices** across 2 physical A100-PCIE-40GB GPUs:

| Device Name | Parent GPU | MIG Profile | SMs | Memory |
|---|---|---|---|---|
| `gpu-0-mig-9-4-4` | GPU-0 | 3g.20gb | 42 | ~20096 MiB |
| `gpu-0-mig-5-0-4` | GPU-0 | 4g.20gb | 56 | ~20096 MiB |
| `gpu-1-mig-0-0-8` | GPU-1 | 7g.40gb | 98 | ~40320 MiB |

All devices are published via a single ResourceSlice with `deviceClassName: mig.nvidia.com`.

## 2. Test 11-1a: Server-Side SM Limit (Daemon-Level)

| Property | Value |
|---|---|
| File | `manifests/module11/11-1a-sm-server-side.yaml` |
| CUDA source | `manifests/module11/cuda/fma_bench.cu` |
| Setup | 1 ResourceClaim (MPS, 50% threads), 3 Pods sharing it |
| Target | MIG device via `mig.nvidia.com` |
| Total thread request | 150% (3 × 50%) |

### Design

Three pods share a single MIG device via one MPS-enabled ResourceClaim. The claim uses `defaultActiveThreadPercentage: 50`.

### Expected Behavior

- The scheduler schedules all 3 pods, since DRA treats the claim as a single shared device.
- MPS `ActiveThreadPercentage` is a per-daemon setting, not per-client. All 3 pods share the same 50% limit cooperatively.
- No hard enforcement of per-pod SM limits.

### Actual Results

- All 3 pods were scheduled and reached Running state successfully.
- The claim was allocated to `gpu-0-mig-9-4-4` (3g.20gb, 42 SMs) and shared by all 3 pods via MPS.
- `nvidia-smi` query returned `[Insufficient Permissions]` for memory/utilization fields -- expected for MIG devices under MPS.

**Compute Benchmark Results** (FMA stress test: N=16M, INNER_ITERS=64, 200 launches):

| Pod | Throughput (GFLOPS) | Elapsed (s) |
|---|---|---|
| Pod 1 | 1801.18 | 0.238 |
| Pod 2 | 1807.46 | 0.238 |
| Pod 3 | 1828.38 | 0.235 |

### Conclusions

All 3 pods achieved nearly identical throughput (~1800 GFLOPS). This confirms:

1. `defaultActiveThreadPercentage: 50` is a **per-daemon ceiling**, not per-client. All 3 clients share the same MPS daemon and the same 50% SM limit cooperatively. The MPS daemon serializes kernel launches across all clients, so each pod sees the full 50% ceiling rather than a 50%/3 share.
2. **Server-side `defaultActiveThreadPercentage` cannot provide per-client SM isolation.** This is an MPS architecture limitation — the MPS control daemon applies `ActiveThreadPercentage` globally to all clients under it. There is no server-side mechanism to assign different thread percentages to different clients within the same daemon. To achieve per-client SM control, see 11-1b (client-side env var).
3. **The DRA scheduler is capacity-unaware for MPS thread parameters.** 3 × 50% = 150% total is allowed because the scheduler treats the shared ResourceClaim as a single allocation unit — it does not sum per-client thread requests. Oversubscription detection is delegated entirely to the MPS runtime, which in turn treats the percentage as a shared ceiling rather than a per-client quota.

## 3. Test 11-1b: Client-Side SM Limit (CUDA_MPS_ACTIVE_THREAD_PERCENTAGE)

| Property | Value |
|---|---|
| File | `manifests/module11/11-1b-sm-client-side.yaml` |
| CUDA source | `manifests/module11/cuda/fma_bench.cu` (shared with 11-1a) |
| Setup | 1 ResourceClaim (MPS, daemon at 50%), 3 Pods with client-side env var |
| Target | MIG device via `mig.nvidia.com` |
| Pod thread percentages | Pod 1: 10%, Pod 2: 30%, Pod 3: 50% |

### Design

Since server-side `defaultActiveThreadPercentage` is a per-daemon ceiling (not per-client), this test uses the **client-side** env var `CUDA_MPS_ACTIVE_THREAD_PERCENTAGE` to enforce per-pod SM limits. Three pods share one MIG device, each with a different thread percentage.

### Actual Results

FMA stress test on MIG device with 56 SMs:

| Pod | Thread % | SMs | Throughput (GFLOPS) |
|---|---|---|---|
| Pod 1 | 10% | 4 | 351.49 |
| Pod 2 | 30% | 16 | 1320.80 |
| Pod 3 | 50% | 28 | 2159.12 |

### Throughput Ratio Analysis

| Comparison | Thread % ratio | SM ratio | GFLOPS ratio | Scaling efficiency |
|---|---|---|---|---|
| Pod 2 / Pod 1 | 3.0× (30/10) | 4.0× (16/4) | **3.76×** (1320.80/351.49) | ~94% of SM ratio |
| Pod 3 / Pod 1 | 5.0× (50/10) | 7.0× (28/4) | **6.15×** (2159.12/351.49) | ~88% of SM ratio |
| Pod 3 / Pod 2 | 1.67× (50/30) | 1.75× (28/16) | **1.63×** (2159.12/1320.80) | ~93% of SM ratio |

Throughput scales near-linearly with SM count. The slight sub-linear scaling (~88–94% efficiency) is expected due to memory bandwidth becoming the bottleneck as SM count increases — the FMA kernel is compute-heavy but still incurs memory traffic for the initial load and final store.

### Conclusions

1. **`CUDA_MPS_ACTIVE_THREAD_PERCENTAGE` works on MIG**: The client-side env var successfully enforces per-client SM limits on MIG devices, providing true per-pod resource isolation. Each pod's `cudaGetDeviceProperties().multiProcessorCount` reflects the limited SM count (4, 16, 28), not the full MIG slice's SMs.
2. **Throughput is near-linearly proportional to SM count**: Pod 3 (50%, 28 SMs) achieves 6.15× the throughput of Pod 1 (10%, 4 SMs), closely matching the 7× SM ratio. This confirms the SM limit directly controls compute capacity with high efficiency.
3. **Client-side env var provides per-client isolation that server-side cannot**: Unlike `defaultActiveThreadPercentage` (which applies a shared daemon-level ceiling — see 11-1a), the client-side env var assigns each pod a distinct, independently enforced SM limit. This is the recommended approach for per-pod compute isolation on MIG.

## 5. Summary

| Approach | Granularity | Enforced on MIG? | Use Case |
|---|---|---|---|
| Server-side `defaultActiveThreadPercentage` | Per-daemon (shared ceiling) | Yes, but all clients share equally | Baseline throttling |
| Client-side `CUDA_MPS_ACTIVE_THREAD_PERCENTAGE` | Per-client (per-pod) | **Yes** — proportional SM isolation | Production per-pod SM control |

**Key insight**: The DRA scheduler is capacity-unaware for MPS thread parameters — oversubscription (3 × 50% = 150%) is allowed. To enforce per-pod SM isolation on MIG, use the client-side `CUDA_MPS_ACTIVE_THREAD_PERCENTAGE` env var.
