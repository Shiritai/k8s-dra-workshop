# Module 10.4: MIG x MPS

**Status**: ✅ **Verified**

## 1. Introduction
While MIG provides hard isolation, there are cases where a single MIG slice (e.g., 20GB) is still too much for a single lightweight micro-inference container. **Hybrid Scheduling** combines the best of both worlds:
1.  **MIG**: Provides the high-level hardware boundary (e.g., separating Production from Dev).
2.  **MPS**: Provides a sub-boundary within that MIG slice, allowing multiple containers to share the same physical cores and memory pool concurrently.
3.  **Multi-Instance Concurrency**: Launch independent MPS control daemons for different MIG slices on the same physical GPU simultaneously for maximum flexibility.

---

## 2. Core Concepts

### 2.1 The Hierarchy of Sharing
- **MIG-level**: Hard isolation, dedicated hardware resources.
- **MPS-level**: Spatial sharing, concurrent execution of kernels within one slice.

### 2.2 Orchestration via GpuConfig
Kubernetes DRA handles this complexity by allowing a `GpuConfig` object to be attached to a `ResourceClaim`. This tells the driver to initialize the MPS control daemon specifically for the allocated device.

---

## 3. Implementation

### 3.1 Hybrid Resource Claim
A single claim requests a MIG device but marks it for "MPS Sharing".

```yaml
spec:
  devices:
    config:
    - opaque:
        driver: gpu.nvidia.com
        parameters:
          apiVersion: resource.nvidia.com/v1beta1
          kind: GpuConfig
          sharing:
            strategy: mps
    requests:
    - exactly:
        count: 1
        deviceClassName: mig.nvidia.com
```

### 3.2 Multi-Pod MPS Sharing on MIG
Multiple **Pods** can reference the same `ResourceClaim` to share a MIG slice via MPS. Meanwhile, multiple `ResourceClaims` can be active on the same GPU, each binding to a different MIG slice and independently enabling MPS.

```yaml
# Pod 1 and Pod 2 share the same MIG slice via MPS
resourceClaims:
- name: gpu
  resourceClaimName: m10-4a-claim-v2   # shared claim with MPS config
```

The automation script (`module10-4/run.sh`) tests two cases:
- **Case A**: 2 Pods sharing 1 MIG slice via a single MPS-enabled claim
- **Case B**: 4 Pods on 2 MIG slices (2 claims, each MPS-enabled), proving independent MPS daemons per MIG slice

## 4. Verification

### 4.1 Expected Behavior
**Case A**: Two separate Pods sharing one MIG claim both see the same MIG UUID.
**Case B**: Four Pods on two MIG slices — each pair shares its own MIG UUID, and the two pairs use **different** MIG UUIDs (hardware isolation between groups).

### 4.2 Proof of Concurrent Access
Run the automated verification script:
```bash
./scripts/phase3/module10-4/run.sh
```

## 5. Technical Notes
- **IPC Namespace**: The DRA driver handles the necessary mounts for `/tmp/nvidia-mps` automatically, ensuring both containers can communicate with the MPS control daemon.
- **GPU Utilization**: When both containers run kernels, the aggregate GPU utilization of the slice increases, reflecting true concurrent execution.
