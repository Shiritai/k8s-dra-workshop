# Module 11.2: MIG MPS VRAM Limits

**Status**: ✅ **Verified** (K8s v1.34.0, NVIDIA A100-PCIE-40GB x2, DRA driver v25.8.1)

## 1. Introduction

This module tests whether the DRA driver and MPS runtime enforce per-client VRAM (GPU memory) limits on MIG devices. We compare two approaches:

- **Server-side**: `defaultPinnedDeviceMemoryLimit` in the ResourceClaim's `MigDeviceConfig`
- **Client-side**: `CUDA_MPS_PINNED_DEVICE_MEM_LIMIT` environment variable (per-pod limit)

### Cluster Configuration

All tests run on a cluster with **3 MIG devices** across 2 physical A100-PCIE-40GB GPUs:

| Device Name | Parent GPU | MIG Profile | SMs | Memory |
|---|---|---|---|---|
| `gpu-0-mig-9-4-4` | GPU-0 | 3g.20gb | 42 | ~20096 MiB |
| `gpu-0-mig-5-0-4` | GPU-0 | 4g.20gb | 56 | ~20096 MiB |
| `gpu-1-mig-0-0-8` | GPU-1 | 7g.40gb | 98 | ~40320 MiB |

## 2. Test 11-2a: Server-Side VRAM Limit (defaultPinnedDeviceMemoryLimit)

| Property | Value |
|---|---|
| File | `manifests/module11/11-2a-vram-server-side.yaml` |
| CUDA source | `manifests/module11/cuda/vram_limit.cu` |
| Setup | 1 ResourceClaim (MPS, 4Gi pinned memory limit, 50% threads), 3 Pods sharing it |
| Target | MIG device via `mig.nvidia.com` |
| Pod allocation sizes | Pod 1: 4.5 GiB (4608 MiB), Pod 2: 3 GiB (3072 MiB), Pod 3: 3 GiB (3072 MiB) |

### Design

Test whether `defaultPinnedDeviceMemoryLimit: 4Gi` is actually enforced on MIG devices. Pod 1 requests 4.5 GiB, exceeding the 4 GiB limit. If the limit were enforced, Pod 1 should fail — but if the limit is broken, all three pods should succeed (since the total 10.5 GiB fits within the ~20 GiB physical MIG memory). The allocation sizes are intentionally the same as 11-2b for a direct server-side vs. client-side comparison.

### Actual Results

Allocated to a MIG device with ~20096 MiB physical memory:

| Pod | ALLOC_MIB | Result | `cudaMemGetInfo` total |
|---|---|---|---|
| Pod 1 | 4608 | **SUCCESS** (4.5 GiB allocated + memset OK) | 20096 MiB |
| Pod 2 | 3072 | **SUCCESS** (3 GiB allocated + memset OK) | 20096 MiB |
| Pod 3 | 3072 | **SUCCESS** (3 GiB allocated + memset OK) | 20096 MiB |

### Critical Finding

**`defaultPinnedDeviceMemoryLimit: 4Gi` is NOT enforced on MIG devices.**

Pod 1 successfully allocated 4.5 GiB of VRAM despite the 4 GiB per-client limit. This is 12.5% over the configured limit, and the allocation succeeded without error. The `cudaMemset` call also succeeded, confirming the memory was genuinely usable.

Compare with 11-2b (client-side limit): the same Pod 1 with the same 4.5 GiB allocation **fails** when using client-side `CUDA_MPS_PINNED_DEVICE_MEM_LIMIT=0=4096M`. This confirms that the server-side limit is broken, not that the allocation size is within bounds.

### Conclusions

1. **`defaultPinnedDeviceMemoryLimit` has no effect on MIG devices**: The MPS daemon accepts the configuration, but the CUDA runtime does not enforce it.
2. **`cudaMemGetInfo` reports full physical memory**: All clients see 20096 MiB total — the full MIG partition, not the 4 GiB per-client limit.
3. **Physical MIG memory is the only real constraint**: On MIG devices, the only hard limit is the physical memory of the MIG slice.
4. **Direct comparison with 11-2b proves the bug**: Using the same allocation sizes (4.5/3/3 GiB) with the same 4 GiB limit, 11-2a (server-side) allows all three pods to succeed while 11-2b (client-side) correctly rejects Pod 1.

## 3. Root Cause Analysis: Why MPS Memory Limit Fails on MIG

Investigation of the DRA driver source code (`k8s-dra-driver-gpu v25.8.1`) reveals a fundamental incompatibility between MIG mode and MPS server-side memory enforcement.

### Root Cause — `EXCLUSIVE_PROCESS` Incompatibility

`sharing.go:282` calls `setComputeMode("EXCLUSIVE_PROCESS")` on the parent GPU UUID before deploying the MPS control daemon. On non-MIG GPUs, this ensures the MPS server is the sole GPU context owner, enabling two critical properties:

1. **Reliable per-client memory accounting**: The MPS server is the only CUDA context, so it can precisely track each client's memory usage and enforce `set_default_device_pinned_mem_limit`.
2. **Timing safety**: No external process can create a CUDA context before the daemon sends its configuration commands, so the MPS server always starts with the correct defaults.

However, [NVIDIA MIG documentation](https://docs.nvidia.com/datacenter/tesla/mig-user-guide/latest/) states: *"EXCLUSIVE_PROCESS mode is not supported when the GPU is in MIG mode."* This call **silently fails** on MIG-enabled GPUs — no error, no warning. Without `EXCLUSIVE_PROCESS`, the MPS server cannot enforce per-client memory limits. `set_default_device_pinned_mem_limit` is accepted by the control daemon but has no effect on the CUDA runtime, regardless of config timing.

### Note on Config Timing

The daemon template (`mps-control-daemon.tmpl.yaml`) sends `set_default_device_pinned_mem_limit` after `nvidia-cuda-mps-control -d`. This ordering is correct by design — the MPS server is lazy-spawned (only starts when the first client connects), so the daemon records the default value before any server exists. On non-MIG GPUs with `EXCLUSIVE_PROCESS`, this works reliably. On MIG without `EXCLUSIVE_PROCESS`, a theoretical race condition exists (workload could connect before the limit command), but this is secondary — even with perfect timing, the limit is not enforced.

**Note**: `defaultActiveThreadPercentage` works because it controls the MPS control daemon's thread scheduling policy, which does not depend on `EXCLUSIVE_PROCESS` or memory accounting.

### Source Code References

- `k8s-dra-driver-gpu/cmd/gpu-kubelet-plugin/sharing.go:203-229` — `set_default_device_pinned_mem_limit` invocation
- `k8s-dra-driver-gpu/cmd/gpu-kubelet-plugin/sharing.go:282` — `setComputeMode("EXCLUSIVE_PROCESS")`
- `k8s-dra-driver-gpu/templates/mps-control-daemon.tmpl.yaml:37` — MPS daemon startup sequence

## 4. Test 11-2b: Client-Side VRAM Limit (CUDA_MPS_PINNED_DEVICE_MEM_LIMIT)

| Property | Value |
|---|---|
| File | `manifests/module11/11-2b-vram-client-side.yaml` |
| CUDA source | `manifests/module11/cuda/vram_limit.cu` (shared with 11-2a) |
| Setup | 1 ResourceClaim (MPS, no `defaultPinnedDeviceMemoryLimit` — pure client-side test), 3 Pods with client-side env var |
| Target | MIG device via `mig.nvidia.com` |
| Client limit | `CUDA_MPS_PINNED_DEVICE_MEM_LIMIT=0=4096M` set in each pod's env |
| Pod allocation sizes | Pod 1: 4.5 GiB (4608 MiB), Pod 2: 3 GiB (3072 MiB), Pod 3: 3 GiB (3072 MiB) |

### Design

This test uses the **client-side** env var `CUDA_MPS_PINNED_DEVICE_MEM_LIMIT=0=4096M` to enforce a 4 GiB per-client memory limit. The claim has **no** `defaultPinnedDeviceMemoryLimit` (pure client-side control), making this a clean counterpart to 11-2a. The allocation sizes are intentionally the same as 11-2a for a direct comparison:

- Pod 1: 4.5 GiB alloc (4608 MiB) — should fail (exceeds 4 GiB MPS client limit, but fits physically)
- Pod 2: 3 GiB alloc (3072 MiB) — should succeed (within 4 GiB limit)
- Pod 3: 3 GiB alloc (3072 MiB) — should succeed (within 4 GiB limit)

### Actual Results

Allocated to MIG 3g.20gb, ~20096 MiB physical:

| Pod | ALLOC_MIB | `cudaMemGetInfo` free | Result | Why |
|---|---|---|---|---|
| Pod 1 | 4608 | 3875 MiB | **FAILED** (code 2) | 4608 > 3875 MiB (MPS limit enforced) |
| Pod 2 | 3072 | 3875 MiB | **SUCCESS** (memset OK) | 3072 < 3875 MiB (within limit) |
| Pod 3 | 3072 | 3875 MiB | **SUCCESS** (memset OK) | 3072 < 3875 MiB (within limit) |

### Critical Findings

1. **`CUDA_MPS_PINNED_DEVICE_MEM_LIMIT` works on MIG**: The client-side env var successfully enforces per-client memory limits on MIG devices, unlike the server-side `defaultPinnedDeviceMemoryLimit`.
2. **`cudaMemGetInfo` is virtualized**: All 3 pods report `free=3875 MiB` (≈4096M limit minus ~221 MiB context overhead), **not** the physical ~20096 MiB. The client-side env var virtualizes the CUDA memory query to reflect the per-client limit.
3. **Pod 1 failure is MPS enforcement, not physical OOM**: The device has ~20 GiB physical memory, but Pod 1's 4.5 GiB request is rejected because it exceeds the 4 GiB client limit. Compare with 11-2a where the same 4.5 GiB allocation succeeds with the broken server-side limit.
4. **Pods 2 and 3 coexist successfully**: Both allocate 3 GiB each (6 GiB total), well within both the per-client 4 GiB limit and the ~20 GiB physical limit.

**Single-pod verification** (deployed separately on clean device): A single pod with `CUDA_MPS_PINNED_DEVICE_MEM_LIMIT=0=4096M` on a fresh MIG device saw `free=3875 MiB` (vs ~19761 MiB without the env var). 2 GiB allocation succeeded; 4 GiB allocation failed. This confirms the env var is the sole cause of the limit.

## 6. Summary

| Approach | Enforced on MIG? | `cudaMemGetInfo` virtualized? | Use Case |
|---|---|---|---|
| Server-side `defaultPinnedDeviceMemoryLimit` | **No** (DRA driver bug) | No (reports full physical) | Broken on MIG — do not rely on |
| Client-side `CUDA_MPS_PINNED_DEVICE_MEM_LIMIT` | **Yes** | **Yes** (reports per-client limit) | Recommended workaround for MIG |

**Key insight**: Server-side MPS memory limits are broken on MIG because `EXCLUSIVE_PROCESS` mode — required for MPS server-side memory accounting — is not supported on MIG-enabled GPUs. The DRA driver's `setComputeMode("EXCLUSIVE_PROCESS")` call silently fails, leaving the MPS server unable to enforce per-client memory limits. The daemon startup sequence itself is correct by design (it works on non-MIG GPUs). Tests 11-2a and 11-2b use the same allocation sizes (4.5/3/3 GiB) with the same 4 GiB limit, providing a direct comparison: server-side allows all allocations (broken), while client-side correctly rejects the over-limit request. Use client-side `CUDA_MPS_PINNED_DEVICE_MEM_LIMIT` as the reliable workaround.
