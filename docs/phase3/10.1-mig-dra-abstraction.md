# Module 10.1: MIG x DRA Abstraction

**Status**: ✅ **Verified**
**Integration**: 🔗 **Integrated with Module 1 (Device Discovery)**

## 1. Introduction
In a traditional GPU cluster, users must manually find the UUID of a specific MIG slice. This is error-prone and hard to scale.

**Module 10.1** demonstrates how DRA abstracts this with four progressive levels:
- **Case A**: Request *any* MIG instance — zero knowledge of hardware required
- **Case B**: Request a *specific profile* (`7g.40gb`) using a CEL attribute selector
- **Case C**: Request profile `3g.20gb`
- **Case D**: Request profile `4g.20gb`

---

## 2. Abstraction Mechanism

The NVIDIA DRA driver publishes MIG devices with `type: mig` in the `ResourceSlice`. A `DeviceClass` named `mig.nvidia.com` selects all devices matching this type.

Each MIG device exposes rich attributes for CEL filtering:

| Attribute | Type | Example | Description |
|-----------|------|---------|-------------|
| `profile` | string | `"7g.40gb"`, `"3g.20gb"`, `"4g.20gb"` | MIG profile name |
| `type` | string | `"mig"` | Device type |
| `parentUUID` | string | `"GPU-d675..."` | Parent physical GPU |
| `uuid` | string | `"MIG-9ac8..."` | Unique MIG instance ID |

And capacity fields:

| Capacity | Type | Example | Description |
|----------|------|---------|-------------|
| `memory` | quantity | `40320Mi` | VRAM |
| `multiprocessors` | quantity | `98` | Streaming Multiprocessors |
| `copyEngines` | quantity | `7` | DMA copy engines |

## 3. Implementation

### 3.1 Case A: Any MIG Instance (Pure Abstraction)

No filters. Just request the DeviceClass. DRA handles discovery and binding.

```yaml
apiVersion: resource.k8s.io/v1
kind: ResourceClaim
metadata:
  name: m10-1a-any-mig-claim
spec:
  devices:
    requests:
    - name: mig-req
      exactly:
        count: 1
        deviceClassName: mig.nvidia.com
```

### 3.2 Case B: Profile-Based CEL Selector

Target a specific MIG profile by matching the `profile` attribute.

**CEL Syntax**: `device.attributes['gpu.nvidia.com'].profile == '7g.40gb'`

```yaml
apiVersion: resource.k8s.io/v1
kind: ResourceClaim
metadata:
  name: m10-1b-profile-claim
spec:
  devices:
    requests:
    - name: mig-req
      exactly:
        count: 1
        deviceClassName: mig.nvidia.com
        selectors:
        - cel:
            expression: "device.attributes['gpu.nvidia.com'].profile == '7g.40gb'"
```

### 3.3 Case C: Profile `3g.20gb`

```yaml
selectors:
- cel:
    expression: "device.attributes['gpu.nvidia.com'].profile == '3g.20gb'"
```

### 3.4 Case D: Profile `4g.20gb`

```yaml
selectors:
- cel:
    expression: "device.attributes['gpu.nvidia.com'].profile == '4g.20gb'"
```

> **Key Insight**: The `selectors` field uses `device.attributes['gpu.nvidia.com']` as the domain-qualified namespace, even though the raw ResourceSlice JSON stores attributes at the top level. This is a DRA CEL API convention.

## 4. Verification

### Environment
- GPU 0: MIG enabled with **3g.20gb** + **4g.20gb** profiles
- GPU 1: MIG enabled with **7g.40gb** profile

Run the demo:
```bash
./scripts/phase3/module10-1/run.sh
```

**Expected Results**:
- **Case A**: Pod scheduled to *any* available MIG slice (pure abstraction)
- **Case B**: Pod on `7g.40gb` (98 SM, 40320Mi)
- **Case C**: Pod on `3g.20gb` (42 SM, 20096Mi)
- **Case D**: Pod on `4g.20gb` (56 SM, 20096Mi)

The script runs A first (then cleans up), then B+C+D concurrently — each targets a distinct profile, filling all 3 MIG devices simultaneously.

## 5. CEL Syntax Quick Reference

| Goal | CEL Expression |
|------|---------------|
| Any MIG device | *(no selector needed, DeviceClass handles it)* |
| Profile name | `device.attributes['gpu.nvidia.com'].profile == '7g.40gb'` |
| Specific parent GPU | `device.attributes['gpu.nvidia.com'].parentUUID == 'GPU-xxx'` |
| Architecture | `device.attributes['gpu.nvidia.com'].architecture == 'Ampere'` |
