# Module 10.2: SLA & OOM Isolation for Multi-tenant vLLM

**Status**: ✅ **Verified**

## 1. Introduction
In a multi-tenant AI cluster, "Noisy Neighbors" are a common frustration. If one user's container (e.g., a buggy vLLM instance) attempts to allocate more VRAM than permitted, it can cause the entire GPU to crash or slow down, affecting other tenants.

This module validates that **MIG (Multi-Instance GPU)** provides hardware-level isolation. An OOM event in one slice is strictly contained and does not impact the SLA of other slices on the same physical GPU.

---

## 2. Core Concepts

### 2.1 Hardware Partitioning vs. Software Capping
Unlike standard Kubernetes memory limits (which rely on cgroups), MIG provides dedicated memory controllers and compute engines to each instance. This means cross-instance interference is physically impossible.

### 2.2 SLA-Based Resource Allocation
Using DRA's numerical selectors, we can categorize our infrastructure into different service levels:
- **Platinum Tier (Llama Pod)**: Allocated to a 56 SM (4g.20gb) slice for guaranteed high performance.
- **Standard Tier (Gemma Pod)**: Allocated to a 42 SM (3g.20gb) slice for cost-effective baseline performance.

---

## 3. Implementation

### 3.1 Resource Claims & Pods
The `ResourceClaim` uses `.compareTo()` to enforce the SM-count requirement for each tier.

```yaml
# High SLA (Platinum)
selectors:
- cel:
    expression: "device.capacity['gpu.nvidia.com'].multiprocessors.compareTo(quantity('50')) >= 0"
```

### 3.2 OOM Stress Test
We use a custom CUDA program (`stress.cu`) to attempt an allocation of 25GB on a 20GB slice.

```bash
# Compile and run inside the pod
nvcc stress.cu -o stress
./stress 25
```

---

## 4. Verification

### 4.1 Expected Behavior
1.  **vllm-gemma** (Standard) attempts to allocate 25GB and receives an immediate `CUDA_ERROR_OUT_OF_MEMORY`.
2.  **vllm-llama** (High SLA) continues to function perfectly. Checking `nvidia-smi` inside `vllm-llama` shows 0% impact on its memory or stability.

## 5. Technical Notes
- **Read-Only Manifests**: When using `ConfigMap` to inject code, remember to copy it to a writable directory (e.g., `/tmp/work`) before compiling with `nvcc`.
- **Compute Isolation**: Even if one slice is at 100% compute utilization, the other slice's SMs are physically separate and will not experience "jitter".
