# Module 10.2: Tight Resource Matching (Capacity-Based CEL)

**Status**: ✅ **Verified**
**Integration**: 🔗 **Integrated with Module 10.1 (Attribute Selectors)**

## 1. Introduction

Module 10.1 demonstrated attribute-based selection (string matching on `profile`). But in a heterogeneous MIG environment, you may not know the exact profile names — you just know your workload needs "at least 90 SMs" or "at least 30GB VRAM."

**Module 10.2** demonstrates **capacity-based CEL selectors** using numeric comparison on hardware capacity fields, enabling:
- **Case A**: Selection by SM range 90–100 (positive match, targets 7g.40gb)
- **Case B**: Selection by VRAM size >= 30Gi (positive match, targets 7g.40gb)
- **Case C**: Selection by SM count > 100 (negative match — proves rejection)
- **Case D**: Selection by SM range 50–60 (positive match, targets 4g.20gb)
- **Case E**: Selection by VRAM range 15–25Gi (positive match, targets 3g/4g.20gb)

---

## 2. Core Concepts

### 2.1 Attributes vs. Capacity

DRA devices expose two kinds of filterable fields:

| Kind | Access Pattern | Use Case |
|------|---------------|----------|
| **Attributes** | `device.attributes['gpu.nvidia.com'].profile` | String/version matching (profile name, UUID, architecture) |
| **Capacity** | `device.capacity['gpu.nvidia.com'].memory.compareTo(quantity('30Gi'))` | Numeric comparison (VRAM, SM count, engines) |

### 2.2 The `compareTo` + `quantity()` Pattern

CEL capacity comparisons use the pattern:
```
device.capacity['gpu.nvidia.com'].{field}.compareTo(quantity('{value}')) >= 0
```
- Returns `>0` if device value > threshold
- Returns `0` if equal
- Returns `<0` if device value < threshold

## 3. Implementation

### 3.1 Case A: SM Range Selection

Request a MIG instance with SM count between 90 and 100. The `7g.40gb` profile has **98 SMs**, which falls squarely in this range.

```yaml
apiVersion: resource.k8s.io/v1
kind: ResourceClaim
metadata:
  name: m10-2a-sm-claim
spec:
  devices:
    requests:
    - name: mig-req
      exactly:
        count: 1
        deviceClassName: mig.nvidia.com
        selectors:
        - cel:
            expression: >-
              device.capacity['gpu.nvidia.com'].multiprocessors.compareTo(quantity('90')) >= 0 &&
              device.capacity['gpu.nvidia.com'].multiprocessors.compareTo(quantity('100')) < 0
```

This demonstrates **precise range filtering** on numeric capacity fields.

### 3.2 Case B: VRAM-Based Selection

Request a MIG instance with at least 30Gi of VRAM.

```yaml
apiVersion: resource.k8s.io/v1
kind: ResourceClaim
metadata:
  name: m10-2b-vram-claim
spec:
  devices:
    requests:
    - name: mig-req
      exactly:
        count: 1
        deviceClassName: mig.nvidia.com
        selectors:
        - cel:
            expression: "device.capacity['gpu.nvidia.com'].memory.compareTo(quantity('30Gi')) >= 0"
```

The `7g.40gb` MIG profile has **40320Mi (~39.4Gi)** of VRAM, satisfying the `>= 30Gi` condition.

### 3.3 Case C: SM Threshold (Negative Match)

Request a MIG instance with more than 100 Streaming Multiprocessors. No A100 MIG profile exceeds 100 SMs (the largest, `7g.40gb`, has 98), so the claim **cannot be satisfied** and the Pod stays `Pending`.

```yaml
apiVersion: resource.k8s.io/v1
kind: ResourceClaim
metadata:
  name: m10-2c-sm-range-claim
spec:
  devices:
    requests:
    - name: mig-req
      exactly:
        count: 1
        deviceClassName: mig.nvidia.com
        selectors:
        - cel:
            expression: "device.capacity['gpu.nvidia.com'].multiprocessors.compareTo(quantity('100')) > 0"
```

### 3.4 Case D: SM Range 50–60 (targets 4g.20gb)

```yaml
selectors:
- cel:
    expression: >-
      device.capacity['gpu.nvidia.com'].multiprocessors.compareTo(quantity('50')) >= 0 &&
      device.capacity['gpu.nvidia.com'].multiprocessors.compareTo(quantity('60')) < 0
```

The `4g.20gb` profile has **56 SMs**, matching this range.

### 3.5 Case E: VRAM Range 15–25Gi (targets 3g/4g.20gb)

```yaml
selectors:
- cel:
    expression: >-
      device.capacity['gpu.nvidia.com'].memory.compareTo(quantity('15Gi')) >= 0 &&
      device.capacity['gpu.nvidia.com'].memory.compareTo(quantity('25Gi')) < 0
```

Both `3g.20gb` and `4g.20gb` have **20096Mi (~19.6Gi)** of VRAM, so either can satisfy this claim. This demonstrates range-based filtering that targets a **tier** of profiles rather than a single one.

## 4. Verification

### Environment
- GPU 0: MIG with **3g.20gb** (42 SM, 20096Mi) + **4g.20gb** (56 SM, 20096Mi)
- GPU 1: MIG with **7g.40gb** (98 SM, 40320Mi)

Run the demo:
```bash
./scripts/phase3/module10-2/run.sh
```

The script runs in two groups to avoid resource contention:
- **Group 1**: Cases A+D+E (each targets a different device) + C (negative), all concurrent
- **Group 2**: Case B (needs 7g.40gb freed from A)

**Expected Results**:

| Case | CEL Filter | Target | Result |
|------|-----------|--------|--------|
| A | 90 ≤ SM < 100 | 7g.40gb (98 SM) | ✅ Running |
| B | VRAM >= 30Gi | 7g.40gb (40320Mi) | ✅ Running |
| C | SM > 100 | none | ✅ Pending |
| D | 50 ≤ SM < 60 | 4g.20gb (56 SM) | ✅ Running |
| E | 15Gi ≤ VRAM < 25Gi | 3g/4g.20gb (20096Mi) | ✅ Running |

### Anti-Fragmentation in Practice

With 3 MIG profiles available:
- Cases A and B exclusively target `7g.40gb` (high SM / high VRAM)
- Case D exclusively targets `4g.20gb` (mid SM)
- Case E targets the "20GB tier" (both 3g and 4g)
- No case can "upsize" to a larger profile than needed

This prevents the problem where small workloads consume large MIG slices.
